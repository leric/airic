# Airic Kernel Technical Design v0.1

## 1. Purpose

Airic Kernel is a minimal markdown-configured agent runtime over a user-owned workspace.

The v0.1 goal is to build the smallest useful agent that can:

* Chat with the user.
* Read workspace files.
* Create and edit files.
* Load agent behavior from markdown specs under `.airic/`.
* Load core runtime instructions from a core pack.
* Enter document-type aware editing only when a markdown file explicitly declares `doc_type` in frontmatter.
* Integrate with external UI via ACP (Agent Communication Protocol).

Airic Kernel is not a document management system. It does not own the user’s project documents. It only provides an agent layer over an existing workspace.

---

## 2. Core Principle

The user workspace is owned by the user.

Airic only owns `.airic/`.

```text
user-workspace/
  README.md
  docs/
  src/
  paper/
  notes/
  ...

  .airic/
    config.yml
    specs/
    packs/
    sessions/
    logs/
    cache/
```

Project files, project documents, source code, papers, notes, and drafts remain ordinary user files.

Airic may read and edit them when requested, but it does not impose directory structure, hidden metadata, resource IDs, overlay graphs, or persistent semantic bindings on them.

---

## 3. Non-goals for v0.1

Airic Kernel v0.1 does not implement:

* Extension pack marketplace.
* Background execution agents.
* Decision Queue.
* Review agent.
* Persistent semantic graph.
* Overlay metadata for workspace resources.
* Resource identity tracking.
* Automatic typed links between documents.
* Workflow engine.
* Strict schema or DSL for specs.
* Strong synchronization with user file moves / renames / deletes.

These may be added later, but they are not required to validate the kernel.

---

## 4. `.airic` Directory Layout

Documents in `.airic` are used to control the behavior of Airic Agent, or data generated by Airic Agent.

```text
.airic/
  config.yml

  specs/
    roles/
      thinking-partner.md

    document-types/
      decision.md
      task.md
      note.md

    processes/
      deep-focus.md
      document-editing.md

  packs/
    core/
      base-instruction.md
      roles/
      document-types/
      processes/
    founder/

  sessions/
    <session-id>.json

  logs/
    edits.log

  cache/
    file-index.json
```

### Directory responsibilities

```text
config.yml
  Kernel configuration, including LLM and pack settings.

specs/
  Active markdown specs loaded by the agent runtime.

packs/
  Installed pack content. Packs define reusable agent behavior bundles.
  Core pack provides base instruction and default specs.

sessions/
  Session state and chat metadata.

logs/
  Append-only edit logs and audit trail.

cache/
  Rebuildable cache, such as lightweight file index.
```

### Core Pack Design (v0.1)

Although pack ecosystem is not a v0.1 feature, the structure of the core pack must be defined because it directly affects runtime behavior.

Core pack responsibilities:

* Provide `base-instruction.md` (kernel base instruction).
* Provide minimal default role, document-type, and process specs.
* Serve as the default source of behavior if user does not override.

In v0.1:

* Core pack content can be minimal.
* Kernel loads base instruction from core pack at runtime.
* Specs may be copied or referenced into `.airic/specs`.

## 5. Markdown Spec Documents

All Airic specs are ordinary markdown files with minimal frontmatter.

Spec files are not strict YAML DSLs. Their main content is prose written for the agent to read and follow.

There are three core spec document types:

```text
core.role
core.document-type
core.process
```

### Role Spec

A role spec describes the agent’s behavior mode and thinking style.

Example:

```markdown
---
id: core.thinking-partner
doc_type: core.role
title: Thinking Partner
---

# Thinking Partner

Your job is to help the user think clearly without taking over direction.

## Principles

- Protect the user's deep thinking.
- Ask high-leverage questions only when useful.
- Do not force premature summaries.
- Separate discussion, decision, task, and review.
- Preserve unresolved ideas when they may matter later.
```

### Document-Type Spec

A document-type spec describes what a good document of that type should contain.

Example:

```markdown
---
id: core.decision
doc_type: core.document-type
title: Decision Document
---

# Decision Document

A decision document records a stable decision that should guide future work.

A good decision document should make clear:

- What was decided.
- Why it was decided.
- What alternatives were considered.
- What constraints or consequences follow from the decision.
- What questions remain open.
```

### Process Spec

A process spec describes an optional interaction flow with a relatively clear structure and repeatable steps.

Processes are best suited for interaction patterns that have a recognizable sequence, such as extracting precedents, discovering user stories, or guiding structured analysis.

Example:

```markdown
---
id: core.precedent-extraction
doc_type: core.process
title: Precedent Extraction
---

# Precedent Extraction

This process helps the user extract reusable patterns or precedents from past work or experiences.

## Steps

1. Identify a concrete past case or example.
2. Describe what happened in that case.
3. Analyze why it worked or failed.
4. Extract key patterns or principles.
5. Generalize the patterns into reusable guidance.

## Guidance

- Keep the discussion grounded in specific examples.
- Avoid jumping to abstract conclusions too early.
- Ensure extracted precedents are actionable and transferable.
```

## 6. User Documents and `doc_type`

User project documents are ordinary markdown files in the user workspace.

A user document only participates in document-type aware editing when it explicitly declares `doc_type` in frontmatter.

Example:

```markdown
---
doc_type: core.decision
title: Use markdown-configured kernel
status: draft
---

# Use markdown-configured kernel

## Context

...

## Decision

...
```

The only required field is:

```yaml
doc_type: core.decision
```

Other fields are optional and belong to the user document.

### Invariant

Airic never infers persistent document semantics for user files.

A user file enters document-type aware editing only when it explicitly declares `doc_type` in frontmatter.

## 7. Session Model

A session is a normal chat session with optional state.

```text
Session
  id
  role_id
  current_document: optional
  active_process: optional
  chat_history
  edit_log_refs
```

### `role_id`

Required. Every session must have an active role that defines the agent’s behavior.

The role is selected at startup, either from config or by the user, and can be manually switched during the session.

Airic always loads the corresponding role spec for the active role.

### `current_document`

Optional.

A normal chat session may have no current document.

A current document exists only when the user explicitly opens or creates a document for collaboration.

### `active_process`

Optional.

Processes are not only started via CLI. They can also be activated by the agent through tool calls when appropriate.


### 8.4 Process-Guided Chat

```text
kernel base instruction
+ active role spec

+ chat history
```

---

### 8.5 Process-Guided Document Editing

```text
kernel base instruction
+ active role spec
+ current document content
+ resolved document-type spec, if any
+ active process spec
+ chat history
```

## 9. Context Loading Policy

```text
Always load:
- kernel base instruction (from core pack)
- active role spec
- active process spec, if any
- current document or file, if any
- current document-type spec, if declared
- chat history window
```

Additional context via tools:

```text
- list files
- read file
- search files
- search text
```

## 10. File Operations

The kernel provides minimal file capabilities.

```text
list_files(path)
read_file(path)
create_file(path, content)
edit_file(path, patch)
rename_file(old_path, new_path)
delete_file(path)
search_files(query)
search_text(query)
```

### Editing Policy

For v0.1, file edits should be auditable.

Preferred behavior:

```text
1. Agent proposes an edit.
2. Kernel generates a diff.
3. User confirms or rejects.
4. Kernel writes the file.
5. Edit is recorded in logs/edits.log.
```

Fast-apply mode may be added later, but reviewable diff should be the default.

---

## 11. Runtime Loop

For each user message:

```text
1. Load session state.

2. Resolve active role spec.

3. Resolve active process spec, if active_process exists.

4. If current_document exists:
   - read current file
   - parse markdown frontmatter
   - if frontmatter.doc_type exists:
       resolve corresponding document-type spec

5. Build runtime context.

6. Call LLM.

7. If LLM requests tools:
   - read/search/list files as needed
   - generate file edit patches if needed
   - ask user confirmation before write

8. Apply confirmed edits.

9. Update session state and edit log.

10. Return response to user.
```

## 12. Config

Minimal `.airic/config.yml`:

```yaml
default_role: core.thinking-partner

llm:
  provider: openai
  model: gpt-4o
  temperature: 0.7
  max_tokens: 4096

packs:
  core: .airic/packs/core

spec_paths:
  roles: .airic/specs/roles
  document_types: .airic/specs/document-types
  processes: .airic/specs/processes

editing:
  require_confirmation: true

cache:
  enabled: true
```

## 13. UI Integration (ACP)

Airic Kernel is UI-agnostic.

Interaction layer is decoupled via ACP (Agent Communication Protocol), enabling integration with editors like Zed.

Responsibilities:

```text
Kernel:
  - runtime
  - context building
  - tool execution

UI (ACP client):
  - chat interface
  - file view/edit
  - process visualization
```

---

## 14. MVP Acceptance Criteria

1. Chat with role spec.
2. File editing.
3. doc_type-based editing.
4. Core pack base instruction loading.
5. LLM configurable.
6. ACP-compatible interaction.
7. No hidden document system.

---

## 15. Key Design Invariants

* Workspace ownership
* Spec simplicity
* Explicit doc_type
* Core pack provides base instruction
* Process can be agent-triggered
* No persistent metadata

---

## 16. Summary

Airic Kernel v0.1 is a thin agent runtime over a user-owned workspace.

Core idea:

> Agent behavior is defined by markdown + core pack + runtime context.

MVP:

```text
chat
+ file editing
+ markdown specs
+ core pack base instruction
+ doc_type editing
+ ACP integration
```
